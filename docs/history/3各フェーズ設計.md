## 公開インターフェースと内部データ契約

### 共通型

```ts
type Source =
  | "system_events"
  | "application_events"
  | "os"
  | "display_drivers";

type CoverageStatus =
  | "ok"
  | "no_data"
  | "denied"
  | "unavailable"
  | "failed"
  | "timeout";

type WarningCode =
  | "snapshot_is_collection_time"
  | "future_window_clipped"
  | "source_partial"
  | "evidence_limit_reached"
  | "privacy_review_required";

type NormalizedPlan = {
  problem: string;                  // マスク済み
  incident_time_utc: string;
  original_utc_offset: string;
  before_minutes: number;
  after_minutes: number;
  window_start_utc: string;
  window_end_utc: string;
  sources: Source[];
};

type Coverage = {
  source: Source;
  status: CoverageStatus;
  item_count: number;
  truncated_before: boolean;
  truncated_after: boolean;
};

type Evidence = {
  id: string;
  kind: "event_log" | "operating_system" | "display_driver";
  temporal_kind: "incident_event" | "collection_snapshot";
  timestamp_utc: string;
  source: Source;
  summary: string;                  // 最大160文字
  details: string;                  // 最大2,000文字
};

type PrivacyStats = {
  masked_values: number;
  unsafe_messages_replaced: number;
  dropped_evidence: number;
  review_required: true;
  raw_artifact_written: false;
};
```

`Case` は `schema_version: "1"`、UUID `case_id`、`mode`、実効時間窓を加えた `plan`、`collected_at_utc`、`coverage`、`privacy`、最大200件の `evidence` を持つstrict objectとする。

### MCPツール

- `plan_collection`
  - 入力は問題文1～2,000文字、offset付きRFC3339、前後0～30分、重複なしの1～4 source。前後省略時は各5分、両方0は拒否する。
  - RFC3339をカレンダー日付まで検証し、UTCへ正規化する。plan hashやserver stateは作らない。
  - 出力は `{ plan, warnings }`。
- `collect_incident_window`
  - 入力は `{ plan, mode, fixture_name? }`。fixture時だけ `gpu-driver-reset` を必須とし、live時のfixture名は拒否する。
  - future endを収集開始時刻へ縮める。開始も未来なら開始を同じ時刻へ縮め、空の未来区間を作らない。
  - 出力は `{ case_id, coverage, evidence_index, privacy, warnings }`。indexからdetailsを除く。
- `inspect_evidence`
  - UUID case IDと、重複なし1～20件のevidence IDを受ける。
  - 保存caseを毎回strict再検証し、入力順で `{ case_id, evidence, coverage, warnings }` を返す。
- `export_support_report`
  - `hypotheses` では1～3仮説、連続rank、`low | medium`、実在ID、1～5件の `not_proven` を必須にする。
  - `insufficient_evidence` では仮説0件、missing evidence 1件以上とする。
  - snapshot IDだけを参照する仮説は拒否する。
  - UUID report IDを生成し、`report-<uuid>.md` を `wx` で新規作成する。出力は `{ report_id, markdown, coverage, privacy_review_warning }`。

成功応答はstrict output schemaに従うobjectを `structuredContent` に置き、同一objectの `JSON.stringify` 結果をTextContentへ置く。入力・実行エラーはMCP `isError` と固定された安全なエラーコードだけを返し、入力値、stderr、stack、実パスを反映しない。

### 保存・privacy・Markdown

- MCPのcaseは `%LOCALAPPDATA%\IncidentDocket\cases\<uuid>.json`、reportは同階層の `reports\report-<uuid>.md` に保存する。
- 非Windows fixtureは `os.tmpdir()\IncidentDocket` を使う。MCPには保存先指定を追加しない。
- `demo` は証拠timeline Markdownをstdoutへ出す。`--output` 指定時だけ、マスク済みcase JSONと同じtimeline Markdownを指定ディレクトリへ排他的に保存する。
- allowlist射影後、PC名、user、domain、path、UNC、SID、email、IP、MAC、GUID、credential/token、prompt injectionを固定markerへ置換する。
- 残留検出したfree textは `<REDACTED_UNSAFE_MESSAGE>` へ置換し、再scanにも失敗するevidenceだけをdropする。
- Markdownの動的文字列は改行を空白化し、HTML、backtick、画像、リンク、見出し・表注入に必要な記号をescapeする。
- reportは「Summary」「Coverage」「HypothesesまたはInsufficient evidence」「Missing evidence」「Next steps」「Privacy review」の固定構成にする。

## Phase詳細

### Stage A — 設計・repo固定

- 入力：現行統合計画、空のローカルrepo、GitHubアカウント `4i7`。
- 実装：
  - 本計画を `DESIGN.md` として記載する。
  - `AGENTS.md` にPhase順、npmコマンド、4 source限定、実機ログ禁止、完了Gateを短く記載する。
  - MIT `LICENSE` と、`node_modules/`、`dist/`、coverage、`.tgz`、`.env`、Codex作業用ディレクトリだけを除外する `.gitignore` を追加する。`cases/` や `reports/` はCWD生成を検出できるよう無視しない。
  - branchを `main` に変更し、公開repo `4i7/IncidentDocket` と `origin` を作成する。
- 失敗時：同名remoteに既存内容が現れた場合やpushできない場合は、上書きせず停止する。
- 検証：必須設計内に `IncidentSlice`、pnpm、HTML、WER、Reliability、services、7 sourcesが残っていないことを検索する。期限後バックログでの言及だけは許可する。
- 完了：`docs: define IncidentDocket submission MVP` を最初のcommitとしてpushする。

### Phase 1 — 決定論的fixture core

- 入力：NormalizedPlan、4 sourceのraw fixture row。
- 実装：
  - Node 24、ESM、strict TypeScriptを設定する。runtimeは `@modelcontextprotocol/sdk@1.29.0` と `zod@4.4.3`、devは `typescript@7.0.2`、`vitest@4.1.10`、`@types/node@24.13.3` をexact保存する。
  - Zod strict schema、RFC3339正規化、包含時間窓、future clippingを純粋関数として実装する。
  - incident eventを時刻、固定source順、native key、canonical masked contentでsortする。
  - eventには `EV-001`、OSには `OS-001`、driverには `DR-001` をsort後に付ける。snapshotは収集時刻をtimestampとしtimelineへ混ぜない。
  - fixtureに既定incident `2026-07-16T09:30:00+09:00`、前後5分、4 sourceを持たせ、eventを意図的に未整列で格納する。
- 失敗時：未知field/source、不正日付、重複source、上限超過はcase生成前に拒否する。
- 検証：Z、正負offset、包含境界、両window 0、30分超過、同時刻tie-break、安定ID、snapshot分離、200件上限を `test/core.test.ts` で確認する。
- 完了：PowerShellなしで `npm test` と `npm run build` が成功し、`feat: add deterministic fixture core` をcommitする。

### Phase 2 — privacy・storage・CLI demo

- 入力：Phase 1のfixture rowと正規化plan。
- 実装：
  - allowlist、再帰mask、残留scan、Evidence組立、case保存、Markdown escape、timeline rendererを `src/core.ts` に追加する。
  - fixtureへ架空のuser/path/SID/email/IP/MAC/GUID/token、Markdown/HTML/prompt injectionを追加する。
  - caseは一時ファイルからatomic renameし、既存caseを上書きしない。
  - `src/index.ts` に依存なしの最小argv parserを置き、`incident-docket demo --fixture gpu-driver-reset [--output <directory>]` を実装する。
  - demoは仮説・原因・完成support reportを生成せず、「Incident timeline」「Current state at collection time」「Coverage」「Privacy review」だけをstdoutへ出す。
- 失敗時：任意fixture path、未知option、既存出力ファイル、保存先作成失敗は非ゼロ終了する。stdoutへstackやraw値を出さない。
- 検証：全fake secretと注入文字列がcase、stdout、Markdownから消え、marker、mask件数、drop件数だけが残ることを確認する。path containmentと非上書きもテストする。
- 完了：fixture demoが決定論的なtimelineを生成し、`feat: add privacy-safe evidence pipeline` をcommitする。

### Phase 3 — MCP・GPT-5.6 fixture E2E

- 入力：Phase 2のcore、GPT-5.6が作る仮説入力。
- 実装：
  - STDIO serverと4ツールだけを登録し、固定instructionsを設定する。
  - plan/inspectはread-only、collect/exportは非破壊write、全ツール `openWorldHint: false` とする。UUID reportのためexportは非idempotentとする。
  - case IDから安全にcaseを再読込し、inspect前とreport前に再マスクする。
  - report rendererとUUID `wx` 保存を接続し、同名衝突時だけ最大3回UUIDを再生成する。
  - SDKのin-memory transportでtool一覧、schema、annotations、structured/text一致を自動検証する。
  - Codexへ登録し、GPT-5.6でfixtureのplan→collect→inspect→exportを実行する。
- 失敗時：未知case/ID、case schema破損、rank欠落、高confidence、snapshot-only仮説、不正outcome相関、保存失敗を安全なtool errorにする。
- 検証：unknown ID、insufficient evidence、prompt injection、report Markdown escape、非上書きを自動テストする。E2Eでは固定済みAI回答を使わず、新規のevidence-linked仮説が生成されることを確認する。
- 完了：4 tool flowがGPT-5.6で完走し、`feat: expose IncidentDocket MCP workflow` をcommitする。

### Phase 4 — Windows 11 live collector

- 入力：CasePlanと4 source。
- 実装：
  - sourceごとにPowerShell 5.1を別プロセスで順次実行する。各12秒、全体60秒とし、残り時間がないsourceはtimeout coverageにする。
  - System/Applicationはbefore側26件取得後25件保持、after側も古い順26件取得後25件保持し、`LogName + RecordId` で境界重複を除く。
  - OSは `Win32_OperatingSystem` 1件、driverはDISPLAYだけを安定sortして最大20件取得する。
  - PowerShellはstrict allowlist objectをUTF-8 JSON→Base64→ASCII一行としてstdoutへ返す。`denied`、`unavailable`、`failed` は固定statusだけを返す。
  - Node側はBase64、JSON、source固有schemaを再検証する。PowerShell起動自体が不可能な場合だけtool全体をerrorにする。
  - Windows 11判定は `process.platform === "win32"` かつ `os.release()` のbuild 22000以上とする。昇格要求は行わない。
- 失敗時：単一sourceのdenied、timeout、no data、malformed outputはcoverageへ変換して他sourceを続行する。全source失敗でもcoverage-only caseを保存する。
- 検証：fake runnerで各coverage状態とtimeout継続を自動テストする。実機確認は別terminal/Codex taskで行い、raw stdout、stderr、event本文を主実装タスクやrepoへ渡さない。前後の `git status` でCWD不変を確認する。
- 完了：Windows 11標準ユーザーで部分失敗を含むcaseが生成され、`feat: collect allowlisted Windows evidence` をcommitする。

### Phase 5 — CI・package・clean install

- 実装：
  - `package.json` にNode shebang付き `dist/index.js` のbin、`engines.node >=22`、`files: ["dist","collectors","samples"]`、`build`、`test`、buildのみの `prepack` を設定する。
  - install lifecycle script、formatter、linter、追加runtime dependencyは導入しない。
  - Windows/Node 24の単一GitHub Actions jobで `npm ci`、`npm test`、`npm run build` を実行する。
  - `npm pack --dry-run` の内容をallowlist照合し、source、test、tsconfig、実データ、ローカルpathが入らないことを確認する。
- 失敗時：pack内容超過、clean install後のCLI/MCP失敗、CWD artifact生成はrelease blockerとする。
- 検証：Windows Sandboxまたは空のnpm prefixで `.tgz` のみをglobal installし、fixture demoとMCP handshakeを実行する。TypeScript compilerやsource treeがなくても動くことを確認する。
- 完了：`build: package judge-ready release` をcommitし、同じ主Codexタスクから `/feedback` Session IDを取得する。

## Assumptions and Defaults

- 「Markdownのみ」は公開timeline/report形式を指し、strictな内部case JSON保存は許可する。
- demoはMarkdownをstdoutへ出し、`--output` 指定時だけファイルにも保存する。
- before/after省略時は各5分。sourceの既定値は設けず、明示指定を必須にする。
- plan hash、server-side plan state、content-addressed report、HTML、追加collector、GUIは作らない。
- source実行は並列化せず順次実行する。4×12秒が60秒以内に収まるため、追加schedulerは不要。
- MCP成功応答だけにstructured/text同一契約を適用し、失敗はMCP標準の安全な `isError` とする。
