# IncidentDocket 提出MVP 統合設計・実装計画

## 1. 固定方針

- 提出トラックは `Developer Tools`。
- 主対象はWindowsアプリ・ドライバーを保守する開発者。一般利用者向けとは主張しない。
- 開発環境はNode.js 24、実行要件はNode.js 22以上、package managerはnpm。
- 公開MIT repo `IncidentDocket`、英語README、Codex上のGPT-5.6を推論ホストとして使用する。
- Windows 11 live収集はSystem、Application、OS、display driverの4ソースのみ。
- 出力はMarkdownのみ。HTML、WER、Reliability、services等は期限後バックログへ送る。
- コア実装とテストは同一Codexタスクで進め、最後にその`/feedback` Session IDを取得する。
- `DESIGN.md`を唯一の設計ソース、`AGENTS.md`を実行順・禁止事項・検証コマンドの短い契約とする。旧PLAN 2/3はcommitしない。

製品定義：

> IncidentDocket is a privacy-bounded Windows evidence collector for developers that narrows an incident window, masks known sensitive identifiers, and lets GPT-5.6 produce an evidence-linked issue report through Codex.

## 2. アーキテクチャと公開インターフェース

### 構成

- ESM TypeScript + Windows PowerShell 5.1。
- 主要実装は`src/index.ts`、`src/core.ts`、`collectors/windows.ps1`。
- syntheticデータは`samples/gpu-driver-reset.json`、集約テストは`test/core.test.ts`。
- 直接依存は`@modelcontextprotocol/sdk@1.29.0`と`zod@4.4.3`のみ。hash、UUID、path、process、rendererはNode標準ライブラリを使う。
- `package-lock.json`をcommitし、`npm ci`を再現可能なinstall手順とする。
- `package.json`は`dist`、`collectors`、`samples`だけをpack対象にし、install lifecycle scriptは定義しない。`prepack`はbuildだけを実行する。

### データフロー

```text
Codex / GPT-5.6
  -> plan_collection
  -> collect_incident_window
  -> inspect_evidence
  -> GPT-5.6による仮説生成
  -> export_support_report
  -> privacy review付きMarkdown
```

### MCPツール

1. `plan_collection`

   - 入力：問題文1～2,000文字、offset付きRFC3339時刻、前後0～30分、4ソースの重複なし配列。
   - 前後両方0、offsetなし時刻、未知ソースを拒否。
   - 問題文をマスクし、UTCへ正規化した収集planを返す。server stateとplan hashは持たない。
   - `readOnlyHint: true`、`openWorldHint: false`。

2. `collect_incident_window`

   - 入力：正規化plan、`live | fixture`、fixture時のみ`gpu-driver-reset`。
   - liveはWindows以外で拒否。Windows 11を正式対応環境とする。
   - 各source 12秒、全体60秒。source失敗は`coverage`へ記録し、他sourceを継続する。
   - UUIDの`case_id`、coverage、最大200件のevidence index、mask統計、警告を返す。
   - future endは収集開始時刻まで縮め、不完全windowとして記録する。

3. `inspect_evidence`

   - 入力：UUID `case_id`、重複なしのevidence ID 1～20件。
   - 保存済みcaseをstrict schemaで再検証し、未知ID・path traversalを拒否。
   - 返却前に再マスクし、各detailsを2,000文字以下に制限する。
   - `readOnlyHint: true`。

4. `export_support_report`

   - 入力：`case_id`、`hypotheses | insufficient_evidence`、summary、最大3仮説、missing evidence、next steps。
   - 仮説は連続rank、`low | medium` confidence、実在evidence ID、`not_proven`を必須とする。
   - snapshotだけを障害時点の直接証拠として扱う仮説を拒否する。
   - UUIDのreport IDを作り、`report-<uuid>.md`を排他的新規作成する。既存ファイルは上書きしない。
   - Markdown全文、coverage、privacy review警告を返す。

全ツールは`structuredContent`へschema準拠objectを返し、同じminified JSONをTextContentにも返す。

### Caseと証拠

```text
case:
  schema_version
  case_id
  mode
  plan
  collected_at_utc
  coverage[]
  privacy
  evidence[]

evidence:
  id
  kind
  temporal_kind: incident_event | collection_snapshot
  timestamp_utc
  source
  summary
  details
```

- incident eventは時刻順、同時刻はsource/native key/contentで安定sortする。
- evidence IDはsort後に`EV-001`、`OS-001`、`DR-001`形式で付与する。
- OSとdriverは`collection_snapshot`としてtimelineから分離する。
- case保存先はWindows live/MCPでは`%LOCALAPPDATA%\IncidentDocket\cases`。非Windows fixtureは`os.tmpdir()`を使う。
- CLI fixtureのみ明示`--output`を許可し、MCPは任意保存先を受け取らない。

### Collectorとprivacy

- System/Applicationは`Get-WinEvent -FilterHashtable`でLevel 1～3を取得し、障害前後それぞれ最大25件を保持する。
- OSは`Win32_OperatingSystem`、driverは`Win32_PnPSignedDriver`のDISPLAYのみ。
- PowerShellは許可フィールドだけを`PSCustomObject`へ射影し、UTF-8 JSONをBase64化したASCII一行をstdoutへ返す。
- PC名、username、domain、絶対path、UNC、SID、email、IP、MAC、GUID、credential/tokenを固定markerへ置換する。
- 残留scanで危険な文字列が残ればmessageを置換し、それでも残るevidenceをdropする。raw fallbackはしない。
- raw evidence、raw stderr、stack trace、実ユーザーパスを保存・返却しない。
- 動的文字列はMarkdown escapeし、script、image、link、HTML、backtick注入を無効化する。
- 全reportに「自動マスク後も未知の機密情報が残り得るため共有前確認が必要」と表示する。

CLIは次の2コマンドだけを提供する。

```text
incident-docket mcp
incident-docket demo --fixture gpu-driver-reset [--output <directory>]
```

`demo`はマスク済みcaseとtimelineまでを生成し、仮説や完成reportを固定出力しない。

## 3. 作業フェーズとGate

### Stage A — 個別フェーズの詳細設計

1. `DESIGN.md`へ本仕様、API、schema、privacy境界、package、提出要件を記載する。
2. 各実装Phaseの入力、出力、失敗時動作、テスト、完了条件を同じ文書内で固定する。
3. `AGENTS.md`へnpmコマンド、実装順、対象外、実機ログをcommitや主Codexタスクへ渡さない規則を書く。
4. `main`、MIT、`.gitignore`、公開remoteを準備し、`docs: define IncidentDocket submission MVP`として最初のcommitを作る。

Gate：未確定事項、旧名称、pnpm、HTML、7ソースへの参照が設計の必須項目に残っていない。

### Stage B — 通常実装とテスト

1. Core・fixture

   - npm/TypeScript/Vitest基盤、schema、時刻正規化、stable sort、UUID、fixtureを実装。
   - `feat: add deterministic fixture core`。
   - Gate：PowerShellなしでfixture testとbuildが成功。

2. Privacy・storage・Markdown

   - allowlist、mask、残留scan、case保存、Markdown renderer、CLI demoを実装。
   - `feat: add privacy-safe evidence pipeline`。
   - Gate：fixture内のfake user/path/IP/token/injectionがcase・stdout・Markdownに残らない。

3. MCP・GPT-5.6 E2E

   - 4ツール、instructions、annotations、structured outputを実装。
   - Codexへ登録し、GPT-5.6でplan→collect→inspect→exportを完走。
   - `feat: expose IncidentDocket MCP workflow`。
   - Gate：固定済みAI回答なしで新規仮説とevidence-linked reportが生成される。

4. Windows live collector

   - System、Application、OS、display driverを実装。
   - denied、timeout、no dataをcoverageへ変換し、標準ユーザーで確認。
   - `feat: collect allowlisted Windows evidence`。
   - Gate：CWDにlive artifactを作らず、source単位の失敗でもcaseを生成する。

5. Package・CI・clean install

   - Windows/Node 24の単一CIで`npm ci`、test、buildを実行。
   - `.tgz`に必要ファイルだけが含まれることを確認し、clean環境でglobal installする。
   - `build: package judge-ready release`。
   - Gate：sourceとTypeScript compilerなしでfixture demoとMCP起動が成功。

全コア作業は同一Codexタスクで行い、Phaseごとにcommitする。Stage B完了後に`/feedback` Session IDを取得する。

### Stage C — README・小さな修正・提出

- 英語READMEへproblem、developer audience、install、direct Release URL、supported platform、fixture/live、privacy、test、Codex/GPT-5.6の役割、人間の判断、対象外を記載する。
- Devpost英語説明、公開YouTube動画、Release `.tgz`を作成する。
- 動画は2分45～55秒。build/pack実演ではなく、問題、GPT-5.6 tool flow、マスク済み証拠、evidence-linked report、Codexでの開発方法を見せる。
- Feature freeze後はinstall、privacy、demo、提出導線のblockerだけを修正する。各bugは共有処理のroot causeを直し、最小regression testを追加する。
- README記載コマンドをコピー実行し、ログアウト状態でrepo、Release、YouTube、Devpost URLを確認する。
- `docs: prepare OpenAI Build Week submission`を最終commitとし、tag `v0.1.0`を作る。

## 4. テストと受入条件

- Schema：未知field/source、文字数、配列上限、offsetなし時刻、両window 0を拒否。
- Time：`Z`、`+09:00`、`-07:00`、包含境界、future clipping、前後上限。
- Privacy：user、path、SID、email、IP、MAC、GUID、token、prompt injection、残留時drop。
- Evidence：stable fixture IDs、sort、snapshot分離、unknown ID、UUID/path traversal。
- Report：hypotheses/insufficient相関、rank、confidence、unknown ID、Markdown injection、非上書き。
- MCP：tool数4、stdout純粋性、structured/text一致、annotations、source timeout。
- Windows：denied、no data、timeout、日本語・emoji、CWD無変更。
- Package：`npm ci`、test、build、pack dry-run、clean `.tgz` install、CLI demo、MCP接続。
- Submission：GPT-5.6表示、実tool call、公開YouTube、3分未満、音声、英語、README、Session ID。

必須検証コマンド：

```powershell
npm ci
npm test
npm run build
npm pack --dry-run
npm pack
npm install --global .\incident-docket-0.1.0.tgz
incident-docket demo --fixture gpu-driver-reset
codex mcp add incident_docket -- incident-docket mcp
codex --model gpt-5.6
```
